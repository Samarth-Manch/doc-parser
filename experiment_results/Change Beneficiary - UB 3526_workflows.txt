
====================================================================================================
DETAILED WORKFLOW REPORT
====================================================================================================

INITIATOR WORKFLOW (3 steps)
----------------------------------------------------------------------------------------------------

Step 1: Login: The ASM team logs in to the Manch application using the provided credentials.
  Action Type: login

Step 2: Single Transaction:

Step 3: The ASM team can start a new transaction by clicking Create > Onboarding request > select the template and enter all the details( refer below mentioned table) and submit
  Action Type: submit

SPOC WORKFLOW (10 steps)
----------------------------------------------------------------------------------------------------

Step 1: SMS Notification: An SMS containing a link is sent to the SPOC’s mobile number.
  Action Type: notify

Step 2: OTP Authentication:
  Action Type: verify

Step 3: The SPOC opens the link and receives an OTP (One-Time Password) for additional authentication.
  Action Type: verify

Step 4: The SPOC enters the OTP, and after successful verification, the Manch platform opens the link.
  Action Type: verify

Step 5: Outlet Code Entry:  The SPOC enters the Outlet Code, which is validated by the Manch system against the code associated with the generated link.
  Action Type: validate

Step 6: Add a new question “Do you want to change your current beneficiary?”

Step 7: Values : Yes and No

Step 8: Agreement to Policies:

Step 9: Upon accessing the link, the SPOC is prompted to agree to the data privacy policy and terms of conditions (Change beneficiary -same TOC (outlet kyc).

Step 10: Form Opening: Manch will open the form and SPOC needs to enter the required details as mentioned below

APPROVER WORKFLOW (51 steps)
----------------------------------------------------------------------------------------------------

Step 1: Assignment to Approver: Once all the mandatory fields are entered, validated, and the request is submitted, the system automatically assigns the request to the appropriate approver.
  Action Type: validate

Step 2: Individual Approval (Without Group concept) :
  Action Type: approve

Step 3: The approval flow for individuals will only be routed to the TSE approvers in two specific cases, followed by the UB team as mentioned below.
  Action Type: approve

Step 4: In all other scenarios, it directly goes to the UB team.

Step 5: Establishment Approval (Without Group concept): All requests will it directly goes to the UB team.
  Action Type: approve

Step 6: Group approval: All requests will first go to the ASM and then proceed to the UB team.
  Action Type: approve

Step 7: Approver access: Approvers have two methods to access the transaction, review and act on requests as mentioned below -
  Action Type: approve

Step 8: Email and SMS Notification:
  Action Type: notify

Step 9: The Manch system sends an email and SMS notification to the approver with a link to review the request.
  Action Type: approve

Step 10: The approver clicks the link to access all request details and can either approve or reject the request based on the provided information.
  Action Type: approve

Step 11: Manch System Login:
  Action Type: login

Step 12: The approver can log in to the Manch system, navigate to the "Pending Requests" section, and review the details.
  Action Type: login

Step 13: After reviewing, the approver can either approve or reject the request.
  Action Type: approve

Step 14: TSE View and Action

Step 15: Log into the Manch system: TSE logs into the Manch system and views all pending requests.
  Action Type: login

Step 16: Open the transaction: TSE opens a transaction to see the details mentioned below:

Step 17: Outlet code

Step 18: Outlet name

Step 19: Licensee name

Step 20: Beneficiary name

Step 21: Authorization-letter

Step 22: Respond to mandatory question: TSE answers the mandatory question: "Are you approving this request?" by selecting either "Yes" or "No" from the dropdown menu.

Step 23: Approve or Reject the request:
  Action Type: approve

Step 24: If TSE clicks "Approve," It will submit. The request then moves to the next level of approver.
  Action Type: approve

Step 25: If TSE clicks "Reject," they select a rejection reason from a dropdown and submit it. The request returns to the SPOC for necessary modifications.
  Action Type: reject

Step 26: Resubmit by SPOC : Once the SPOC submits the required details, the request assigns back to TSE, who can then repeat the approval or rejection process as outlined above.
  Action Type: approve

Step 27: ASM View and Action

Step 28: Log into the Manch system: ASM logs into the Manch system and views all pending requests.
  Action Type: login

Step 29: Open the transaction: ASM opens a transaction to see the details uploaded in the bulk file, details entered by the SPOC, along with the relevant documents.
  Action Type: upload

Step 30: Approve or reject the request:
  Action Type: approve

Step 31: If ASM clicks "Approve," It will submit. The request then moves to the next level of approver.
  Action Type: approve

Step 32: If ASM clicks "Reject," they select a rejection reason from a dropdown (configured based on UB rejection reasons) and submit it. The request returns to the SPOC for necessary modifications
  Action Type: reject

Step 33: Resubmit by SPOC : Once the SPOC submits the required details, the request assigns back to ASM, who can then repeat the approval or rejection process as outlined above.
  Action Type: approve

Step 34: UB approver View and Action
  Action Type: approve

Step 35: Log into the Manch system: UB approver logs into the Manch system and views all pending requests.
  Action Type: login

Step 36: Open the transaction:

Step 37: Individual Use case :

Step 38: UB approver opens a transaction to see the details uploaded in the bulk file, and details entered by the SPOC, along with the relevant documents.
  Action Type: upload

Step 39: Group Use case :

Step 40: UB approver opens a transaction to see the details uploaded in the bulk file, and details entered by the SPOC, along with the relevant documents.
  Action Type: upload

Step 41: UB approver sees group member details along with uploaded documents and match score.
  Action Type: upload

Step 42: Approve or reject the request:
  Action Type: approve

Step 43: If UB approver clicks "Approve," It will submit. The request then moves to Accepted status.
  Action Type: approve

Step 44: If UB approver clicks "Reject," they select a rejection reason from a dropdown (configured based on UB rejection reasons) and submit it. The request returns to the SPOC for necessary modifications.
  Action Type: approve

Step 45: Resubmit by SPOC :
  Action Type: submit

Step 46: Once the SPOC submits the required details, the request assigns back to TSE(In case of individual use cases) or ASM (In case of group use cases) who can then repeat the approval or rejection process as outlined above
  Action Type: approve

Step 47: If UB team rejects it will assign back to SPOC and enter the details
  Action Type: reject

Step 48: Once SPOC resubmits, it will be reassigned to ASM / TSE, and ASM /TSE needs to take the actions mentioned in the ASM / TSE section.
  Action Type: submit

Step 49: Once ASM resubmits, it will be reassigned to UB Team, and UB Team needs to take the actions mentioned in the UB Team section.
  Action Type: submit

Step 50: Once UB team submit the transaction, it will be in Accepted status.
  Action Type: submit

Step 51: If any of the approvals again rejects the re-submitted outlets information then it will go through the same flow as mentioned above.
  Action Type: approve
