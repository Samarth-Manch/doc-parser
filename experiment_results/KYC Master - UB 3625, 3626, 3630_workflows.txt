INITIATOR WORKFLOW (20 steps)
----------------------------------------------------------------------------------------------------

Step 1: Single Transaction :

Step 2: UB team will initiate the single transaction by clicking on the create button and selecting the template as Group member outbound process.
  Action Type: create

Step 3: UB team needs to enter the below mentioned details in the first PANEL named “group owner details”.

Step 4: Based on this group owner outlet code, the system will auto derive the below mentioned details in a non-editable format

Step 5: Next panel will be show as Group member leaving details as mentioned below.

Step 6: UB team will select the Group member outlet code from the dropdown. If there are multiple group member leaving from the group, they have an option + by clicking on this option they can add n, number of group members details.

Step 7: Once they select all the group member then they can submit the transaction.
  Action Type: submit

Step 1: UB central team should update the EDV for all the new group member details before starting this process.
  Action Type: update

Step 2: Login: The Central Team Member team logs in to the Manch application using the provided credentials.
  Action Type: login

Step 3: Bulk Transaction:

Step 4: The UB Central Team Member can start a new transaction by clicking Bulk  > Bulk Upload > select the “GROUP MEMBER INBOUND” process and upload the bulk file having below mentioned details.
  Action Type: upload

Step 5: Bulk format will have below mentioned columns

Step 6: Group Owner Outlet Code

Step 7: ASM Name

Step 8: ASM Email ID
  Action Type: notify

Step 9: ASM Mobile Number

Step 10: Based on the group owner outlet code in the bulk file the system will auto derive the below mentioned details in a non-editable format

Step 11: Next panel named as Group member confirmation details. System will derive the newly added group member details based on the type as New mentioned in EDV.

Step 12: The Group Member Confirmation Details will include all newly added outlets to the group.

Step 13: Once all the validations are completed then system will submit the request and it will be assigned to SPOC.
  Action Type: validate

APPROVER WORKFLOW (45 steps)
----------------------------------------------------------------------------------------------------

Step 1: System will assign this transaction to the entered ASM.

Step 2: System will show all the details as mentioned in the previous steps.

Step 3: In the group member leaving details panel, system will ask below mentioned question

Step 4: Are the confirming that below mentioned group members are leavling the group?

Step 5: Dropdown Values : Yes / No

Step 6: System will ask for the Inactivation date also.

Step 7: Approve or reject the request:
  Action Type: approve

Step 8: If ASM clicks "Approve," It will submit. The request then moves to Accepted status.
  Action Type: approve

Step 9: If ASM clicks "Reject," they select a rejection reason from a dropdown (configured based on UB rejection reasons) and submit it. The request returns to the central team for necessary modifications.
  Action Type: reject

Step 10: Once ASM approve this request, System will update the status as inactive with inactivation date for all the group members.
  Action Type: approve

Step 11: Manch will generate one report for this process. UB team to share the report format.

Step 12: Login: The UB team logs in to the Manch application using the provided credentials.
  Action Type: login

Step 13: Single Transaction: The UB team can start a new transaction by clicking Create > Onboarding request > select the template [Master Status Update Process] and enter all the details (refer below mentioned table) and submit
  Action Type: submit

Step 14: After entering all the details, UB team will click on Submit and the transaction will be Accepted and the status of respective outlet code will be updated in the Master.
  Action Type: submit

Step 1: Assignment to Approver: Once all the mandatory fields are entered, validated, and the request is submitted, the system automatically assigns the request to the appropriate approver.
  Action Type: validate

Step 2: All requests will first go to the ASM and then proceed to the UB team.

Step 3: Approver access: Approvers have two methods to access the transaction, review and act on requests as mentioned below -
  Action Type: approve

Step 4: Email and SMS Notification:
  Action Type: notify

Step 5: The Manch system sends an email and SMS notification to the approver with a link to review the request.
  Action Type: approve

Step 6: The approver clicks the link to access all request details and can either approve or reject the request based on the information provided.
  Action Type: approve

Step 7: Manch System Login:
  Action Type: login

Step 8: The approver can log in to the Manch system, navigate to the "Pending Requests" section, and review the details.
  Action Type: login

Step 9: After reviewing, the approver can either approve or reject the request.
  Action Type: approve

Step 10: ASM View and Action:

Step 11: Log into the Manch system: ASM logs into the Manch system and views all pending requests.
  Action Type: login

Step 12: Open the transaction: ASM opens a transaction to see the details uploaded in the bulk file, details entered by the SPOC, along with the confirmation of newly added group member question.
  Action Type: upload

Step 13: Approve or reject the request:
  Action Type: approve

Step 14: If ASM clicks "Approve," It will submit. The request then moves to the next level of approver (UB central team).
  Action Type: approve

Step 15: If ASM clicks "Reject," they select a rejection reason from a dropdown (configured based on UB rejection reasons) and submit it. The request is returned to the SPOC for necessary modifications
  Action Type: reject

Step 16: Resubmit by SPOC : Once the SPOC submits the required details, the request assigns back to ASM, who can then repeat the approval or rejection process as outlined above.
  Action Type: approve

Step 17: UB Central Team Approver View and Action:
  Action Type: approve

Step 18: Log into the Manch system: UB Approver logs into the Manch system and views all pending requests.
  Action Type: login

Step 19: Open the transaction:

Step 20: UB approver opens a transaction to see the details uploaded in the bulk file, and details entered by the SPOC, along with the relevant documents.
  Action Type: upload

Step 21: UB approver sees group member details along with uploaded documents and match score.
  Action Type: upload

Step 22: Approve or reject the request:
  Action Type: approve

Step 23: If UB approver clicks "Approve," It will submit. The request then moves to Accepted status.
  Action Type: approve

Step 24: If UB approver clicks "Reject," they select a rejection reason from a dropdown (configured based on UB rejection reasons) and submit it. The request returns to the SPOC for necessary modifications.
  Action Type: approve

Step 25: Resubmit by SPOC :
  Action Type: submit

Step 26: Once the SPOC submits the required details, the request assigns back to ASM, who can then repeat the approval or rejection process as outlined above.
  Action Type: approve

Step 27: If any of the approvals again reject the re-submitted outlets’ information, then it will go through the same flow as mentioned above.
  Action Type: approve

Step 28: One report will be created for this process. UB team needs to share the format.
  Action Type: create

Step 29: UB team needs to provide the SMS format for Group member inbound process.
  Action Type: notify

Step 30: UB team needs to provide the Reminder service SMS format for Group member inbound process.
  Action Type: notify

Step 31: UB team needs to provide the both processes report formats.

SPOC WORKFLOW (16 steps)
----------------------------------------------------------------------------------------------------

Step 1: SMS Notification: An SMS containing a link is sent to SPOC’s mobile number.
  Action Type: notify

Step 2: OTP Authentication:
  Action Type: verify

Step 3: The SPOC opens the link and receives an OTP (One-Time Password) for additional authentication.
  Action Type: verify

Step 4: The SPOC enters the OTP, and after successful verification, the Manch platform opens the link.
  Action Type: verify

Step 5: Group Owner Outlet Code Entry:  The SPOC enters the Group owner Outlet Code, which is validated by the Manch system against the code associated with the generated link.
  Action Type: validate

Step 6: Agreement to Policies:

Step 7: Upon accessing the link, the SPOC is prompted to agree to the data privacy policy and terms of conditions (same Terms as outlet KYC process).

Step 8: Form Opening:

Step 9: SPOC will open the form by clicking on enter details, and SPOC needs to enter the required details as mentioned below.

Step 10: System will show the Group owner details along with The Group Member  Details that includes all newly added outlets to the group in below mentioned format.

Step 11: If there are more than 1 Group member then system will show all the group member in the same format.

Step 12: System will ask the below mentioned confirmation question to SPOC –

Step 13: Are the confirming that below mentioned group members are newly added in the group?

Step 14: Dropdown Values: Yes / No

Step 15: Once SPOC selects the dropdown value as Yes, then new PANEL will open as Group Member Documents.

Step 16: The system will ask for the group member documents as mentioned in the table below.
