INITIATOR WORKFLOW (5 steps)
----------------------------------------------------------------------------------------------------

Step 1: Login: The UBL team logs in to the Manch application using the provided credentials.
  Action Type: login

Step 2: File Upload:
  Action Type: upload

Step 3: The UBL team uploads a bulk file containing all necessary details such as outlet codes, contact information, and any other required data.
  Action Type: upload

Step 4: The system validates the uploaded file for completeness.
  Action Type: upload

Step 5: Below are the data point mentioned for the file upload format -
  Action Type: upload

SPOC WORKFLOW (12 steps)
----------------------------------------------------------------------------------------------------

Step 1: SMS Notification: An SMS containing a link is sent to the SPOCâ€™s mobile number.
  Action Type: notify

Step 2: SMS Reminder: Manch will share the SMS containing a link again to SPOC mobile number on twice a Weekly basis till the KYC is submitted by SPOC.
  Action Type: submit

Step 3: Note: Days of the week needs to be finalized.
  Notes: ['Note: Days of the week needs to be finalized.']

Step 4: OTP Authentication:
  Action Type: verify

Step 5: The SPOC opens the link and receives an OTP (One-Time Password) for additional authentication.
  Action Type: verify

Step 6: The SPOC enters the OTP, and after successful verification, the Manch platform opens the link.
  Action Type: verify

Step 7: Outlet Code Entry:  The SPOC enters the Outlet Code, which is validated by the Manch system against the code associated with the generated link.
  Action Type: validate

Step 8: Agreement to Policies:

Step 9: Upon accessing the link, the SPOC is prompted to agree to the data privacy policy and terms of conditions.

Step 10: Note: The wording of the terms of conditions and data privacy policy is provided by the UB team.
  Notes: ['Note: The wording of the terms of conditions and data privacy policy is provided by the UB team.']

Step 11: This can be a hyperlink redirecting the user to view the terms of conditions

Step 12: Form Opening: Manch will open the form and SPOC needs to enter the required details as mentioned below

APPROVER WORKFLOW (63 steps)
----------------------------------------------------------------------------------------------------

Step 1: Assignment to Approver: Once all the mandatory fields are entered, validated, and the request is submitted, the system automatically assigns the request to the appropriate approver.
  Action Type: validate

Step 2: Individual Approval (Without Group concept) :
  Action Type: approve

Step 3: The approval flow for individuals will only be routed to the TSE approvers in two specific cases, followed by the UB team as mentioned below.
  Action Type: approve

Step 4: In all other scenarios, it directly goes to the UB team.

Step 5: Establishment Approval (Without Group concept): All requests will it directly goes to the UB team.
  Action Type: approve

Step 6: Group approval: All requests will first go to the ASM, and then proceed to the UB team.
  Action Type: approve

Step 7: Approver access: Approvers have two methods to access the transaction, review and act on requests as mentioned below -
  Action Type: approve

Step 8: Email and SMS Notification:
  Action Type: notify

Step 9: The Manch system sends an email and SMS notification to the approver with a link to review the request.
  Action Type: approve

Step 10: The approver clicks the link to access all request details and can either approve or reject the request based on the provided information.
  Action Type: approve

Step 11: Manch System Login:
  Action Type: login

Step 12: The approver can log in to the Manch system, navigate to the "Pending Requests" section, and review the details.
  Action Type: login

Step 13: After reviewing, the approver can either approve or reject the request.
  Action Type: approve

Step 14: TSE View and Action

Step 15: Log into the Manch system: TSE logs into the Manch system and views all pending requests.
  Action Type: login

Step 16: Open the transaction: TSE opens a transaction to see the details mentioned below:

Step 17: Outlet code

Step 18: Outlet name

Step 19: Licensee name

Step 20: Beneficiary name

Step 21: Authorisation-letter

Step 22: Respond to mandatory question: TSE answers the mandatory question: "Are you approving this request?" by selecting either "Yes" or "No" from the dropdown menu.

Step 23: Approve or Reject the request:
  Action Type: approve

Step 24: If TSE clicks "Approve," It will submit. The request then moves to the next level of approver.
  Action Type: approve

Step 25: If TSE clicks "Reject," they select a rejection reason from a dropdown (configured based on UB rejection reasons) and submit it. The request returns to the SPOC for necessary modifications.
  Action Type: reject

Step 26: Resubmit by SPOC : Once the SPOC submits the required details, the request assigns back to TSE, who can then repeat the approval or rejection process as outlined above.
  Action Type: approve

Step 27: ASM View and Action

Step 28: Log into the Manch system: ASM logs into the Manch system and views all pending requests.
  Action Type: login

Step 29: Open the transaction: ASM opens a transaction to see the details uploaded in the bulk file, details entered by the SPOC, along with the relevant documents.
  Action Type: upload

Step 30: Respond to mandatory question: ASM answers the mandatory question: "Are you approving this group construct?" by selecting either "Yes" or "No" from the dropdown menu.

Step 31: Approve or reject the request:
  Action Type: approve

Step 32: If ASM clicks "Approve," It will submit. The request then moves to the next level of approver.
  Action Type: approve

Step 33: If ASM clicks "Reject," they select a rejection reason from a dropdown (configured based on UB rejection reasons) and submit it. The request returns to the SPOC for necessary modifications.
  Action Type: reject

Step 34: Resubmit by SPOC : Once the SPOC submits the required details, the request assigns back to ASM, who can then repeat the approval or rejection process as outlined above.
  Action Type: approve

Step 35: UB approver View and Action
  Action Type: approve

Step 36: Log into the Manch system: UB approver logs into the Manch system and views all pending requests.
  Action Type: login

Step 37: Open the transaction:

Step 38: Individual Use case :

Step 39: UB approver opens a transaction to see the details uploaded in the bulk file, and details entered by the SPOC, along with the relevant documents.
  Action Type: upload

Step 40: Group Use case :

Step 41: UB approver opens a transaction to see the details uploaded in the bulk file, and details entered by the SPOC, along with the relevant documents.
  Action Type: upload

Step 42: UB approver sees group member details along with uploaded documents and match score.
  Action Type: upload

Step 43: Respond to mandatory question:

Step 44: Individual Use case :

Step 45: UB approver answers the mandatory question: "Are you approving this request?" by selecting either "Yes" or "No" from the dropdown menu.
  Action Type: approve

Step 46: Group Use case :

Step 47: UB approver answers the mandatory question: "Are you approving this group KYC?" for group owner level by selecting either "Yes" or "No" from the dropdown menu.
  Action Type: approve

Step 48: UB approver answers the mandatory question: "Are you approving this group member document?" for each group member level by selecting either "Yes" or "No" from the dropdown menu.
  Action Type: approve

Step 49: Approve or reject the request:
  Action Type: approve

Step 50: If UB approver clicks "Approve," It will submit. The request then moves to Accepted status.
  Action Type: approve

Step 51: If UB approver clicks "Reject," they select a rejection reason from a dropdown (configured based on UB rejection reasons) and submit it. The request returns to the SPOC for necessary modifications.
  Action Type: approve

Step 52: Resubmit by SPOC :
  Action Type: submit

Step 53: Once the SPOC submits the required details, the request assigns back to TSE(In case of individual use cases) or ASM (In case of group use cases) who can then repeat the approval or rejection process as outlined above
  Action Type: approve

Step 54: If UB team rejects X outlets out of Y, and approve Y-X outlets then SPOC will be able view only X outlets (Rejected Outlets). SPOC will be able to edit X outlet details along with documents and re submit. SPOC will not be able to edit other outlets present in the group (X -Y outlets).
  Action Type: approve

Step 55: Once SPOC resubmits, it will be reassigned to ASM / TSE, and ASM /TSE needs to take the actions mentioned in the ASM / TSE section.
  Action Type: submit

Step 56: Once ASM resubmits, it will be reassigned to UB Team, and UB Team needs to take the actions mentioned in the UB Team section.
  Action Type: submit

Step 57: Once UB team submit the transaction, it will be in Accepted status.
  Action Type: submit

Step 58: If any of the approvals again rejects the re-submitted outlets information then it will go through the same flow as mentioned above.
  Action Type: approve

Step 59: Outlet master - Resolved

Step 60: Outlecode to TSE, ASM and state mapping - Resolved

Step 61: Group construct mapping

Step 62: Confirmation on filed level information for SPOC and approver level
  Action Type: approve

Step 63: Need information of UB team members
