Overfitting Risk Report

Assessment of how well the doc-parser BUD-to-API pipeline would generalize to net-new Business Understanding Documents outside the 4 BUDs it was built against (PH Creation, Pidilite Extension, Pidilite Block-Unblock, Vendor Creation Sample).

Scope: .claude/agents/mini/, dispatchers/agents/, static resources Tested BUDs: 4 Mini agents: 12 Pipeline stages: 0–11
62%
Overfit · High

Pipeline is production-ready for India/SAP vendor & material onboarding — but leaks that domain in ~8 places.

The mini-agent prompts are structurally generic: they consume $FIELDS_JSON, $REFERENCE_TABLES, and rule schemas at runtime, and contain no hardcoded field names. Overfitting lives almost entirely in two other places: (1) the dispatchers that glue the pipeline together, and (2) the static keyword tree and reference schemas used as grounding data. A net-new BUD in the same domain (Indian vendor/material procurement) will work; anything further away will degrade sharply.

prompts — mostly generic dispatchers — hardcoded sections 4.5.1/4.5.2 keyword tree — built from 1 BUD reference schemas — 1 per BUD prompt examples — India/SAP jargon

Overfit Scope — BUD-level vs Group-level

The headline 62% is a blended score. For planning "what happens when my next BUD lands", it's more useful to split the overfit into two scopes: risk that's tied to one specific BUD vs risk that's tied to a family of BUDs (Indian Manch procurement). They behave very differently.

22%
BUD-level risk

Assumptions tied to one specific BUD file — things you have to redo mechanically each time. Low correctness risk, purely operational friction.

Offenders at this scope
  • archive/output/complete_format/ — one schema file per BUD, 4 named after existing BUDs
  • run_stages_*.sh — per-BUD script, hardcoded BUD= path
  • convert_to_api_format.py:~1020 — default bud_name = "Vendor Creation"
Failure mode

Missing schema file → Stage 8 crashes immediately. Missing script → you can't invoke the pipeline. Both are noisy, easy to notice, quick to fix (copy an existing one, edit the path). The pipeline itself is LLM-driven and structurally agnostic to field names / panel names.

52%
Group-level risk

Assumptions tied to the family of BUDs — Indian procurement on the Manch platform, written in a specific document template. These fail silently on any in-family BUD whose template or vocabulary drifts.

Offenders at this scope
  • session_based_dispatcher.py:62-115 — literal "4.5.1" / "4.5.2"
  • fix_mandatory_fields.py:143-146 — same section lock-in
  • keyword_tree.json — India KYC branches only (PAN/GSTIN/MSME/CIN/TAN/FSSAI)
  • Rule-Schemas.json — 27 India-KYC rules, no geography tag
  • parser.py:47-55 — tse, asm, spoc, vendor role vocab
  • edv / validate_edv dispatchers — "Sheet1", "Table X.Y" regex
  • FIRST_PARTY / SECOND_PARTY — only session params recognized
  • expression_rule_agent.md + siblings — worked examples saturated with SAP/India vocab
Failure mode

Silent. Stage 7 produces zero session rules. Stage 9 skips rows. EDV stages drop reference tables in non-Sheet1 sheets. No crash, no warning — you only notice when someone reviews the output and says "half the rules are missing".

Read this scope split as

Even within the existing 4 BUDs, the group-level assumptions already don't hold uniformly — run_stages_vendor_extension.sh:60 and run_stages_sunpharma_material.sh:60 both contain the comment # Stage 7 (Session Based) is skipped for this BUD. That's evidence the 4.5.x template lock-in has already broken on 3 of 4 BUDs in-family; the pipeline just worked around it by skipping a whole stage. Any new BUD inherits the same fragility.

Per-Stage Risk Map

Walk-through of each of the 12 pipeline stages with an overfit score, what causes it, and the exact file:line offenders. Stages 0 and 9–11 are deterministic post-processing; stages 1–6 are LLM dispatchers; 7 is optional.

STAGE 0Field Extraction
45% · High

Invoked inline in every run_stages_*.sh and writes directly to archive/output/complete_format/<name>-schema.json. Parser classifies tables via FIELD_HEADER_PATTERNS, INITIATOR_KEYWORDS, SPOC_KEYWORDS, APPROVAL_KEYWORDS — the last three lean on vendor-procurement vocabulary.

  • doc_parser/parser.py:47-55 — keyword lists include tse, asm, spoc, vendor, second party (Pidilite-specific sales roles + procurement jargon)
  • run_stages_ph_creation.sh:31-40 — each BUD needs its own bespoke script; output path hardcoded per BUD
  • archive/output/complete_format/ — exactly 4 BUD-named schemas (ph_creation, sunpharma_material, vendor_extension, raw_material_creation) + 7 numeric-ID fallbacks
STAGE 1Rule Placement
55% · Moderate

The dispatcher itself is clean (dynamic panel loop, no hardcoded fields). But it seeds the LLM with rule_extractor/static/keyword_tree.json — a 491-line tree whose _meta.source_bud literally declares it was built from Vendor Creation Sample BUD.docx. The entire VERIFY L1 subtree lists only India identifiers (PAN, GSTIN, MSME, CIN, TAN, FSSAI, Aadhaar, Driving Licence).

  • rule_extractor/static/keyword_tree.json:5 — "source_bud": "Vendor Creation Sample BUD.docx" is hardcoded
  • keyword_tree.json:23-95 — 9 of 19 L1/VERIFY children are India-specific validations with no equivalent branch for SSN/VAT/EIN
  • keyword_tree.json:29-34 — ordinal maps (e.g. {"Panholder title": 1, "Firstname": 2, ...}) are frozen to the Vendor Creation schema
  • 01_rule_type_placement_agent_v2.md:44-48 — variableName format examples ("Company Code", "Vendor Name", "IFSC Code", "Process Type") all leak SAP/India procurement
STAGE 2Source / Destination
20% · Low

Structurally clean. Dispatcher loads Rule-Schemas.json dynamically, loops panels agnostically. The prompt tells the agent to ignore fields it can't resolve. Main residual risk is that the prompt's abstract JSON examples never show non-English/non-procurement field names — a mild framing bias.

  • source_destination_dispatcher.py — no hardcoded fields or panels detected
  • 02_source_destination_agent_v2.md:60-194 — examples abstract (FIELD_NAME_1, __fieldname1__) — low bias
STAGE 3EDV Rules
40% · Moderate

The dispatcher feeds embedded Excel tables to the agent, but reads them from a hardcoded sheet_name = "Sheet1" and matches table references with a regex that only accepts "Reference Table X.Y" / "Table X.Y" wording. Prompt examples name Pidilite and SAP tables.

  • edv_rule_dispatcher.py:~115 — sheet_name = "Sheet1" hardcoded; ignores additional sheets
  • edv_rule_dispatcher.py:~140 — regex r'(?:reference\s+)?table\s+(\d+\.?\d*)' — fails if BUD calls a lookup "Countries lookup" or "Appendix A"
  • 03_edv_rule_agent_v2.md:33 — example ddType ["VC_VENDOR_TYPES"] (Pidilite)
  • 03_edv_rule_agent_v2.md:272-287 — sample reference tables use Language Key, Country/Region Key, Purchase Organization, "India Domestic", "International"
STAGE 4Validate EDV
42% · Moderate

Inherits the same Sheet1 + regex constraints as Stage 3. DROPDOWN_TYPES is a fixed set; custom types in a new BUD are dropped silently. The prompt's worked example is a PIN-CODE / COMPANY_CODE_PURCHASE_ORGANIZATION walkthrough — strong SAP-India anchor.

  • validate_edv_dispatcher.py:~96 — sheet_name = "Sheet1"
  • validate_edv_dispatcher.py:~191-194 — DROPDOWN_TYPES frozen at {DROPDOWN, EXTERNAL_DROP_DOWN_VALUE, MULTI_DROPDOWN}
  • 04_validate_edv_agent_v2.md:286-302 — sample uses Pin Code → City → District → State → Country
  • 04_validate_edv_agent_v2.md:370-400 — worked example COMPANY_CODE_PURCHASE_ORGANIZATION (SAP schema)
STAGE 5Expression Rules
60% · High

Structurally the dispatcher is excellent — fully dynamic, panel-agnostic, uses context_optimization to slim payloads. The risk lives almost entirely in expression_rule_agent.md: its EXECUTE Rule Placement section teaches the LLM by showing ~20 worked examples, and every single example uses India/SAP vocabulary. This biases the model's judgment on ambiguous non-procurement logic.

  • expression_rule_agent.md:184-196 — placement table uses Region/Country, GST Present, PAN Type, Cheque Image/IFSC, Account Group = INDS/FIVN (SAP vendor account groups)
  • expression_rule_agent.md:220-248 — Copy To example uses _companycodebasicdetails_, _buildingnumberpanandgstdetails_ (BUD-specific variableNames)
  • expression_rule_agent.md:253 — derivation example literally says "If X = India then value is 'DOM IN', else 'INT'"
  • expression_rule_agent.md:414-418 — validation example regex is the Indian PAN pattern /^[A-Z]{5}[0-9]{4}[A-Z]$/
STAGE 6Inter-Panel Rules
45% · Moderate

Two-pass design (detect → classify → resolve) is architecturally sound. Dispatcher is mostly dynamic, but brittle in how it normalizes the detection agent's output — if field names drift, normalization silently drops refs. Prompt examples are saturated with "Basic Details / Bank Details / Vendor Basic Details / Process Type = India-Domestic".

  • inter_panel_dispatcher.py:~152-167 — VALID_CLASSIFICATIONS is a hardcoded set; unknown types silently dropped
  • inter_panel_dispatcher.py:~425-490 — 60+ lines of defensive normalization suggest the detection output format is known-fragile
  • 09_inter_panel_analysis_agent.md:239-318 — all 4 worked examples use "Basic Details → Address Details / Bank Details" with "Process Type = India-Domestic"
  • 09_inter_panel_complex_agent.md:256-371 — derivation example: vendorname → firstname; derivation example: India-Domestic → Local / International
STAGE 7Session Based
75% · Critical

The single worst offender in the pipeline. Entire dispatcher is structured around two specific BUD section numbers: 4.5.1 Initiator Behaviour and 4.5.2 Vendor Behaviour. If a new BUD re-numbers or renames these sections (or omits them), the dispatcher silently produces no session-based rules. Also hardcodes FIRST_PARTY/SECOND_PARTY as the only session types. Every BUD so far happens to skip Stage 7 except PH Creation — because the others don't have 4.5.x. That's overfitting visibly in action.

  • session_based_dispatcher.py:7-8 — docstring asserts "Section 4.5.1 → FIRST_PARTY, 4.5.2 → SECOND_PARTY" as the contract
  • session_based_dispatcher.py:62-64 — filter: "4.5.2" in table.context
  • session_based_dispatcher.py:113-115 — filter: "4.5.1" in table.context
  • session_based_dispatcher.py:38 — magic synthetic field RULE_CHECK_VARIABLE = "__rulecheck__"
  • session_based_dispatcher.py:409-416 — hardcoded rule names "Make Visible - Session Based (Client)" etc. — will miss any BUD that spells them differently
  • run_stages_vendor_extension.sh:60 — comment literally says "Stage 7 (Session Based) is skipped for this BUD"
  • 08_session_based_agent.md:155-204 — all prompt examples use Vendor Type / PAN Number / Organization Name
STAGE 8Convert to API Format
55% · High

Deterministic but structurally coupled. Runs in "injection mode" — --schema archive/output/complete_format/<bud>-schema.json — which means every BUD needs a pre-built schema file. It also assumes a fixed API envelope ['template']['documentTypes'][0]['formFillMetadatas'] and matches fields by panel + name; duplicate field names within a panel collide silently.

  • convert_to_api_format.py:~511 — schema['template']['documentTypes'][0]['formFillMetadatas'] — KeyError if schema shape differs
  • convert_to_api_format.py:~524-544 — panel-scoped name match; 2 identically-named fields in one panel → wrong ID assignment
  • convert_to_api_format.py:564 — comment: "the BUD's 4.5.2 SPOC table"
  • convert_to_api_format.py:~1020 — default bud_name = "Vendor Creation"
  • archive/output/complete_format/*.json — 11 schemas; new BUD requires a 12th to exist before pipeline runs
STAGE 9Fix Mandatory / Editable
70% · Critical

Same 4.5.1/4.5.2 section lock-in as Stage 7. Reads the BUD directly and walks tables, but only writes back mandatory/editable for rows it finds inside sections named 4.5.1 or 4.5.2. Header detection is also brittle — expects literal header strings like "field name" / "field type".

  • fix_mandatory_fields.py:135 — state machine: # States: None, '4.5.1', '4.5.2'
  • fix_mandatory_fields.py:143-146 — literal if '4.5.1' in text / elif '4.5.2' in text
  • fix_mandatory_fields.py:~88-92 — header-string match for column detection
  • fix_mandatory_fields.py:297 — warning: "Skipped N fields not in 4.5.1 or 4.5.2 (kept as-is)"
STAGE 10Resolve EDV Variable Names
22% · Low

Deterministic JSON rewrite. Only assumption is that EDV-related actions match a fixed set {ext_value, external_drop_down, validate_edv} and that variableNames use the __…__ wrapper convention. Survives new BUDs as long as the action set doesn't grow.

  • resolve_edv_varnames.py:28 — TARGET_ACTIONS hardcoded; new EDV rule types would be silently ignored
STAGE 11Post Trigger Linker
15% · Low

Purely structural — scans formFillRules for SERVER-processing rules and links their IDs. No BUD-specific assumptions.

Prompt-Level Domain Bias

The mini-agent prompts themselves are well-parameterized — inputs and instructions are abstract, and they run on any panel JSON. The bias enters through the worked examples. LLMs anchor heavily on example vocabulary: when every illustrative field is PAN Number, Vendor Type, or Process Type = India-Domestic, the model learns to "see" that shape of logic and may miss unfamiliar ones. Remediation — Prioritized

Ordered by impact per effort. Fixing items 1–3 would drop the overall risk score from ~62% to ~35% and make the pipeline usable on any Indian BUD without per-BUD scripts.

1 · Unlock sections 4.5.1 / 4.5.2

Replace literal "4.5.1" / "4.5.2" matching in session_based_dispatcher.py and fix_mandatory_fields.py with a discovery step: scan the BUD for any table whose header/context mentions "Initiator" / "Vendor" / "SPOC" / "Second Party" regardless of section number. Expose --initiator-section / --vendor-section as fallbacks.

Effort · 2-3 days · Impact: very high

2 · Generic run_stages.sh entry point

Collapse the three run_stages_*.sh files into one run_stages.sh that takes --bud and derives a slug from the filename. Stage 0 should auto-generate the complete_format/<slug>-schema.json if absent; stages 7, 9 should warn (not skip silently) when 4.5.x is missing.

Effort · 1 day · Impact: high

3 · Domain-tag the keyword tree & rule schemas

Add a geographies: ["IN"] tag to the 27 India-KYC rules in Rule-Schemas.json and the 9 India-only VERIFY children in keyword_tree.json. Stage 1 dispatcher should filter by an --geographies flag. Ship empty stub branches for US/EU so contributors know where to extend.

Effort · 3 days · Impact: high

4 · De-bias prompt examples

Rewrite worked examples in expression_rule_agent.md, 03_edv_rule_agent_v2.md, 04_validate_edv_agent_v2.md, 08_session_based_agent.md, 09_inter_panel_* to use at least 2 domain-neutral scenarios alongside the SAP-vendor one. The Indian PAN regex in expression_rule_agent.md:414 should become a generic regex pattern illustration.

Effort · 1 day · Impact: moderate

5 · Discover sheets, don't assume "Sheet1"

edv_rule_dispatcher.py:~115 and validate_edv_dispatcher.py:~96 both hardcode the first sheet. Loop over all sheets in each embedded xlsx; emit each as a candidate reference table. Also soften the "Table X.Y" regex — accept any heading/caption near a table.

Effort · 1 day · Impact: moderate

6 · Harden convert_to_api_format field matching

convert_to_api_format.py matches fields by panel-name + field-name and crashes on unfamiliar schema shapes. Add (a) ordinal-position fallback, (b) explicit duplicate-name detection with loud warnings, (c) try/except around the template envelope walk with actionable error messages.

Effort · 2 days · Impact: moderate

7 · Parser keyword generalization

parser.py APPROVAL_KEYWORDS should drop tse/asm or move them to a client-overrides file. Add supplier, counterparty, business partner as SPOC synonyms. Add signatory, authorizer as initiator synonyms.

Effort · 0.5 days · Impact: low

8 · Add an eval harness for "unseen" BUDs

The eval/ framework already compares generated output to reference. Add a dedicated "holdout" mode: pick one BUD, re-run the full pipeline, score it against human reference. Track this metric over time — any new prompt/dispatcher change that lowers holdout accuracy is a regression signal for overfitting.

Effort · 3 days · Impact: long-term high
