Chain of Thought: How This Classification Was Built

This section documents the analysis methodology used to classify all 182 rules into meaningful functional buckets.

Explored Rule-Schemas.json Structure

First, I examined the rules/Rule-Schemas.json file which contains all 182 rule definitions. Each rule has key properties:

  • id - Unique rule identifier
  • name - Human-readable rule name
  • action - What the rule does (OCR, VERIFY, COPY_TO, VALIDATION, etc.)
  • source - Data source type (PAN_IMAGE, GSTIN, EXTERNAL_DATA_VALUE, etc.)
  • processingType - SERVER or CLIENT execution
  • destinationFields - Fields that receive extracted/computed values
  • params - Configuration parameters for the rule

Analyzed Document JSON Outputs

Examined actual document templates in documents/json_output/ to understand how rules are used in practice:

  • Found rules are stored in formFillRules array within each form field
  • Discovered common action types: COPY_TO (39), EXECUTE (38), OCR (21), VERIFY (21), MAKE_INVISIBLE (29)
  • Identified that EDV rules are used for IFSC code lookups to get bank name/branch
  • Noted FFD (Form Fill Dropdown) rules for cascading dropdown population

Reviewed Expression Rules Documentation

Read the Expression Eval Custom Functions-2.pdf to understand the expression rules DSL:

  • Uses expr-eval JavaScript library for custom expressions
  • Functions like vo(id), mvi(), ctfd() for conditional logic
  • Supports complex multi-condition scenarios not covered by standard rules
  • Client-side execution for fast response

Identified Primary Classification Dimensions

Based on analysis, I identified these key dimensions for classification:

  • By Function: What the rule does (extract, validate, copy, control UI)
  • By Domain: What type of data it works with (identity, business, financial)
  • By Processing: Where it runs (server vs client)
  • By Data Source: Where data comes from (OCR, EDV, FFD, transaction)

Grouped Rules by Functional Similarity

Created initial groupings based on the action field patterns:

  • OCR actions → Document data extraction bucket
  • VERIFY actions → Split into Identity vs Business vs Financial validation
  • COPY_TO variations → Data copy, party details, transaction attributes
  • MAKE_* actions → Form control (visibility, mandatory, enable/disable)
  • EXT_VALUE/EXT_DROP_DOWN → EDV and FFD buckets

Created Domain-Specific Sub-Classifications

Further refined validation rules into sub-categories:

  • Identity Validation: PAN, Aadhaar, Voter ID, DL, Passport, RC
  • Business Validation: GSTIN, CIN, MSME, FSSAI, TAN, Drug License
  • Financial Validation: Bank Account, eInvoice, Section 206AB
  • AI/ML: ClassifyImage, Compare Face, Speech To Text, Sentiment

Identified EDV vs FFD Pattern

Key insight from document analysis:

  • EDV (External Data Value): Global lookup tables shared across templates (e.g., IFSC codes, state/city lists)
  • FFD (Form Fill Dropdown): Template-specific dropdown options
  • Both support cascading dropdowns but at different scopes
  • EDV validation can check values against master data (e.g., validate IFSC exists)

Added Keywords and Use Cases for LLM Search

For each bucket, I added:

  • Keywords: Terms an LLM might encounter when user describes their need
  • Use When: Clear guidance on when to use this bucket
  • Bucket Selection Guide: Maps common needs to appropriate buckets

Created Machine-Readable JSON Output

Generated rules_buckets.json for programmatic LLM consumption with:

  • Structured bucket definitions with rules arrays
  • Keyword lists for semantic matching
  • Bucket selection guide for direct need-to-bucket mapping
  • Expression function documentation for custom logic needs

Rule Types Overview

The Manch platform supports two main types of rules:

Standard/Pre-defined Rules

182 server-side or client-side rules with specific, pre-built functionality. Each rule has defined inputs, outputs, and behavior.

Expression Rules

Custom JavaScript-like expressions using expr-eval library for complex conditional logic not covered by standard rules.

Expression Examples

// Make field visible if dropdown equals specific value
mvi(vo("_dropdown_") == "Option1", "_targetField_");

// Copy value based on condition
ctfd(vo("_panType_") == "Company", "CO", "_companyType_");

// Validate age is 18+
adderr(vo("_age_") < 18, "Must be 18 or older", "_age_");

// Conditional visibility based on multiple conditions
mvi(vo("_type_") == "A" and vo("_status_") == "Active", "_field1_", "_field2_");

// Make field mandatory for approver only
mm(mt() == "APPROVER", "_approvalNotes_");

Processing Types

SERVER

Rules executed on server. Full validation capability but requires network round-trip. Used for OCR, validation against external APIs, and database operations.

CLIENT

Rules executed on client device. Faster response, works offline, but limited access to server data. Used for UI control (visibility, mandatory, enable) and expressions.
